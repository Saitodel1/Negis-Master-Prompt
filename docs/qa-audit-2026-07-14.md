# QA, security and reliability audit - 2026-07-14

Scope: live production `https://crm.negis.online`, deployed API server, Supabase-facing frontend, migrations and source. No product data was changed during this audit.

## Evidence collected

- All primary owner routes opened: dashboard, booking, reception, sales, tasks, chat, marketplace, ads, automations and admin.
- Browser console contained no warnings or errors in the checked owner session.
- Production network returned a `400` from the bookings query in Ads and `406` from ad report cache lookup.
- Typecheck passed for `@workspace/negis` and `@workspace/api-server`.
- There is no project test suite, API authorization test suite, RLS role matrix, E2E suite, or load test.

## Critical

### SEC-01 - Employee administration API is public

- Where: `artifacts/api-server/src/app.ts:29-33`, `artifacts/api-server/src/routes/employees.ts:40-143`.
- Reproduce: call `POST /api/admin/employees` on the deployed API server without an Authorization header. An empty body gets domain validation (`400`) instead of `401`; the route is mounted without authentication and uses the Supabase service role.
- Why: the application mounts all routers behind open CORS and no JWT or organization-role middleware.
- Impact: an attacker knowing an organization UUID can create, edit or delete employees and potentially create privileged users.
- Fix: require Supabase JWT, resolve organization from caller membership, allow owner/manager only, and never trust `clinic_id` from the client.

### SEC-02 - Public password reset can change a known user's password

- Where: `artifacts/api-server/src/routes/auth-reset.ts:40-67`.
- Reproduce: public `POST /api/auth/reset-password` reaches input validation without authentication.
- Why: the route enumerates users through the service-role API and directly sets a temporary password.
- Impact: account takeover and denial of service.
- Fix: delete this custom reset path; use Supabase recovery links, rate limiting and anti-abuse controls.

### SEC-03 - Lead webhook accepts unsigned data for any organization

- Where: `api/leads/webhook.ts:31-99`, `api/leads/webhook/[clinicId].ts:34-100`.
- Reproduce: a POST with a known organization UUID and phone is accepted by the handler; there is no per-organization secret, HMAC, replay check, or rate limit.
- Why: the endpoint writes using service role after checking only `clinicId` and `phone`.
- Impact: lead spam, broken analytics, automation abuse and costs.
- Fix: generate a separate random webhook secret per integration connection; require HMAC/signature and timestamp, add dedupe/rate limits/audit logs.

### SEC-04 - WhatsApp webhook does not validate Meta signatures

- Where: `api/webhook.ts:53-108`.
- Reproduce: POST processing has no `X-Hub-Signature-256` verification; GET verify token only protects subscription verification.
- Why: body is trusted before authenticity is established.
- Impact: forged inbound messages can trigger AI work and outbound WhatsApp replies.
- Fix: validate the raw request body with Meta app secret before parsing; add message idempotency and rate limiting.

### SEC-05 - Task RLS grants every member full write access

- Where: `migrations/010_tasks_departments_mvp.sql:147,166-192`; UI-only checks in `artifacts/negis/src/pages/Tasks.tsx:109,363-366`.
- Reproduce: a regular organization member can call Supabase directly and update/delete task records in their organization.
- Why: policies grant `FOR ALL` to any organization member and rely on frontend role checks.
- Impact: employee can reassign, alter or delete tasks outside their role.
- Fix: operation-specific RLS: only owner/manager can create/delete/reassign, assignee can update allowed execution fields, watcher gets read/comment access; add RLS regression tests for owner/manager/employee/watcher.

## High

### SEC-06 - Advertising OAuth and access tokens reach the browser

- Where: `artifacts/negis/src/pages/Ads.tsx:700-705,1152-1153`, `artifacts/negis/src/pages/AdsCallback.tsx:78-121`, `artifacts/api-server/src/routes/ads-tiktok.ts:11-96`, `artifacts/negis/src/lib/facebook-ads.ts:10-74`.
- Reproduce: OAuth state is just an organization id; callback accepts it and returns provider access token to browser. Existing tokens are selected by frontend and Facebook calls put tokens in query strings.
- Impact: token theft, cross-organization OAuth binding and provider-account takeover.
- Fix: server-only OAuth callback/token exchange/storage; signed one-time state bound to user/session; vault or `integration_secrets`; provider calls with Authorization header only.

### SEC-07 - The client exposes the bot verification secret

- Where: Marketplace chatbot settings in live UI and corresponding marketplace code.
- Reproduce: open Market -> Negis Chatbot; the verification token is rendered in a textbox.
- Impact: anyone with organization access can copy a webhook secret; screenshot and browser history leaks become easy.
- Fix: never render the value after creation; show masked state and rotate action. Keep secret server-side only.

### SEC-08 - Wazzup webhook permits query-string/static secret and default organization fallback

- Where: `supabase/functions/wazzup-webhook/index.ts:4-35`.
- Why: a shared key can travel in a URL and unresolved channels may be routed to a default organization.
- Impact: replay/data-crossing risk.
- Fix: no URL secrets; provider-signed requests with timestamp; reject unknown channel, no default organization fallback.

### DATA-01 - Ads module uses a nonexistent booking field in production

- Where: `artifacts/negis/src/pages/Ads.tsx:373-375,551-555`.
- Reproduce: open Ads. Production calls `bookings?...booking_time=gte...` and Supabase returns `400`.
- Why: the actual booking model uses `date` and `time`, as used in `Dashboard.tsx:50-51` and other booking pages.
- Impact: attribution/funnel booking totals are silently wrong.
- Fix: query the real schema consistently or introduce a canonical timestamp column plus migration; surface request failures in the UI.

### DATA-02 - Advertising cache treats "no row" as an error

- Where: `artifacts/negis/src/pages/Ads.tsx:642-648`.
- Reproduce: cache lookup uses `.single()`; no report row returns `406` in production.
- Impact: noisy network errors and fragile refresh path.
- Fix: use `.maybeSingle()` and handle the normal cache-miss state.

### DATA-03 - Booking slot capacity and duplicate detection are client-side races

- Where: `artifacts/negis/src/pages/Booking.tsx:21,321,347,508`; duplicate checks in `Sales.tsx:385,443`.
- Reproduce: two concurrent tabs can both see free capacity / no duplicate and insert.
- Impact: overbooked slots and duplicate customers/leads.
- Fix: transactional Supabase RPC or database constraints: canonical phone, unique lead rule where appropriate, atomic slot capacity check.

### DATA-04 - Task creation is not atomic

- Where: `artifacts/negis/src/pages/Tasks.tsx:237-352`.
- Why: task, checklist, watchers, event, notification, chat card and upload are separate client-side calls.
- Impact: half-created tasks, orphaned files and missing notifications after a network failure or double click.
- Fix: one RPC/Edge Function transaction with idempotency key; upload only after task exists and remove failed uploads.

### DATA-05 - Automation and AI jobs have no robust idempotency/quota

- Where: `migrations/012_automation_runner.sql:175,283`, `supabase/functions/ai-run/index.ts:26-64`, `supabase/functions/_shared/ai.ts:24`.
- Impact: concurrent runners can duplicate actions; any organization member can repeatedly create billable AI calls with client-selected model.
- Fix: idempotency keys/unique run records, role checks, server allowlist of models, per-user and per-organization quotas, backoff and retry policy.

## Medium

### REL-01 - Dashboard has split sources and can show placeholder metrics

- Where: `artifacts/negis/src/pages/Dashboard.tsx:28,101-106`; API endpoint returns fixed/placeholder metrics in `artifacts/api-server/src/routes/dashboard.ts`.
- Why: dashboard combines browser Supabase queries with an unrelated generated API request.
- Impact: revenue and visit metrics can remain zero while operations have data.
- Fix: one audited metric source scoped to organization, server-side aggregation, explicit empty/error states.

### REL-02 - Network errors can leave pages permanently loading

- Where: `Chat.tsx:89-116`; same pattern in `Sales.tsx:203`, `Booking.tsx:267`, `Dashboard.tsx:41`, `Automations.tsx:190`.
- Why: early error return skips `setLoading(false)`.
- Fix: `try/catch/finally` and an error-state component with retry.

### REL-03 - Task notifications are not subscribed in realtime

- Where: `artifacts/negis/src/components/layout/Topbar.tsx:185,217`.
- Impact: a new task can exist but the recipient bell does not update until reload.
- Fix: subscribe to `task_notifications` alongside bookings/automation notifications, clean up channel correctly.

### REL-04 - Data access is over-broad before filtering

- Where: `artifacts/negis/src/pages/Sales.tsx:335-357`.
- Why: loads up to 500 organization bookings client-side then filters by lead.
- Impact: unnecessary exposure/performance cost and broken history at scale.
- Fix: server-side scoped query/RPC with proper indexes and pagination.

### REL-05 - User with several organizations may fail login resolution

- Where: `artifacts/negis/src/contexts/AuthContext.tsx:330`.
- Why: `.single()` is used on user organization roles.
- Fix: select active organization explicitly or use `.maybeSingle()` with a deterministic active-workspace selector.

### PERF-01 - Initial frontend bundle is too large

- Evidence: current bundle roughly 2.0 MB JavaScript, about 585-600 KB gzip.
- Where: eager page imports in `artifacts/negis/src/App.tsx:9`; `xlsx` eagerly imported by Booking/Admin.
- Impact: slow first interaction, especially on mobile connections.
- Fix: route-level `React.lazy`, defer CSV/XLSX library until import/export action, inspect bundle with visualizer and set a budget.

### PERF-02 - Production error overlay is enabled

- Where: `artifacts/negis/vite.config.ts:22`.
- Impact: Replit/dev overlay may leak runtime details in production and adds avoidable client code.
- Fix: enable only in local development.

### UX-01 - Task creation is blocked with no department setup path

- Reproduce: Tasks -> Put task. Department select has no available department; employee selector stays disabled. Empty submit gives no useful actionable guidance.
- Impact: core task workflow cannot start in a new organization.
- Fix: show a clear empty state and owner-only "Create department" link; disable submit with explanation until required fields are valid.

### UX-02 - Inconsistent top bars and inaccessible controls

- Reproduce: on Dashboard the expected global "Add" control is not accessible by name/role; other screens use a different top bar. Profile and task panels are exposed as generic `complementary` areas, not dialogs with focus management.
- Impact: keyboard and screen-reader interaction is unreliable; product behavior changes by page.
- Fix: a single `AppShell/Topbar`, labelled icon buttons, semantic `role=dialog`, focus trap, Escape close, return focus to trigger.

### UX-03 - Mobile layout lacks a verified narrow-screen strategy

- Where: `index.css:440`, `Tasks.tsx:449`, `Chat.tsx:408`.
- Evidence: tables are min-width / desktop-grid based and the app globally hides horizontal overflow, which can hide columns rather than make the interface responsive.
- Fix: define breakpoints, card/table switcher, collapsible filters/sidebar, and test 320/375/768 px in CI.

### UX-04 - Marketplace remains clinic-specific in a universal CRM

- Reproduce: Marketplace copy repeatedly says clinic/patient and shows a medical chatbot prompt even though the platform now supports multiple business verticals.
- Fix: resolve labels/prompts from the selected business vertical and make catalog descriptions neutral by default.

### SEC-09 - Upload validation is client-only

- Where: `Tasks.tsx:340-352`, `010_tasks_departments_mvp.sql:199-207`.
- Impact: arbitrary files may be uploaded by any organization member under a predictable path.
- Fix: storage size/MIME limits, task-level policy, signed downloads, server validation and malware scanning strategy.

## Test gaps that block a "fully stable" claim

- No automated tests for browser flows, role matrix/RLS, API auth, webhook signatures, idempotency, concurrent booking, or billing logic.
- No safe test tenant/accounts for owner, manager, employee and watcher. Live owner session alone cannot prove restrictions work.
- No invoice/payment domain implementation was found matching the requested end-to-end scenario; it cannot be truthfully tested yet.
- Dark theme is not exposed in the inspected interface, so no dark-theme verification was applicable.

## Recommended repair order

1. SEC-01 through SEC-05: close public admin/reset/webhook paths and task RLS before feature work.
2. DATA-01 through DATA-04: restore trustworthy bookings/ads/tasks data.
3. SEC-06 through SEC-09 and DATA-05: integrations, secrets, uploads, AI/automation limits.
4. Reliability/performance/UX items and then add automated coverage so this audit is not a monthly ritual of rediscovering the same holes.
